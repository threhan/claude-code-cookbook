## PR 反馈

高效处理 Pull Request 的审查评论，并通过三阶段错误分析方法从根本上解决问题。

### 使用方法

```bash
# 获取并分析审查评论
gh pr view --comments
“请按优先级对审查评论进行分类，并制定应对计划”

# 详细分析与错误相关的评论
gh pr checks
“请使用三阶段方法分析 CI 错误，并确定根本原因”

# 修复完成后的质量确认
npm test && npm run lint
“修复已完成，请检查回归测试和代码质量”
```

### 基本示例

```bash
# 执行评论分类
gh pr view 123 --comments | head -20
“请将审查评论分为 must/imo/nits/q，并确定处理顺序”

# 收集错误信息
npm run build 2>&1 | tee error.log
“请确定构建错误的根本原因，并提出适当的修复方法”

# 确认修复实现
git diff HEAD~1
“请评估此修复是否妥善解决了审查指出的问题”
```

### 评论分类体系

```
🔴 must: 必须修复
├─ 安全问题
├─ 功能性 bug
├─ 违反设计原则
└─ 违反规范

🟡 imo: 改进建议
├─ 更好的实现方式
├─ 性能改进
├─ 提高可读性
└─ 重构建议

🟢 nits: 轻微指摘
├─ 修正拼写错误
├─ 调整缩进
├─ 添加注释
└─ 微调命名

🔵 q: 提问/确认
├─ 确认实现意图
├─ 明确规格
├─ 设计决策的背景
└─ 探讨替代方案
```

### 三阶段错误分析方法

#### 阶段 1: 信息收集

**必须执行**

- 完全获取错误消息
- 确认堆栈跟踪
- 确定重现条件

**推荐执行**

- 收集环境信息
- 最近的变更历史
- 确认相关日志

#### 阶段 2: 根本原因分析

- 应用 5 Whys 分析
- 追踪依赖关系
- 确认环境差异
- 创建最小可重现代码

#### 阶段 3: 解决方案实施

- 立即处理（热修复）
- 根本解决（本质修复）
- 预防措施（防止复发）

### 应对流程

1. **评论分析**: 按优先级分类
2. **修复计划**: 决定处理顺序
3. **分阶段修复**: Critical → High → Medium → Low
4. **质量确认**: 测试、代码检查、构建
5. **进度报告**: 说明具体的修复内容

### 修复后确认

```bash
# 基本检查
npm test
npm run lint
npm run build

# 回归测试
npm run test:e2e

# 代码质量
npm run test:coverage
```

### 回复模板

**修复完成报告**

```markdown
@reviewer 感谢您的指正。
已修复完成：

- [具体修复内容]
- [测试结果]
- [确认方法]
```

**技术决策说明**

```markdown
实现背景：[原因]
考虑过的替代方案：[选项和决策依据]
采用方案的优点：[优点]
```

### 注意事项

- **遵守优先级**: 按 Critical → High → Medium → Low 的顺序处理
- **测试先行**: 修复前确认回归测试
- **明确报告**: 具体描述修复内容和确认方法
- **建设性对话**: 基于技术依据进行礼貌的沟通
