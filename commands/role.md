## 角色

切换到特定角色，执行专业的分析或工作。

### 使用方法

```bash
/role <角色名> [--agent|-a]
```

### 选项

- `--agent` 或 `-a` : 作为子代理独立执行（建议在进行大规模分析时使用）
  - 使用此选项时，如果角色的描述中包含促进自动委派的短语（如“use PROACTIVELY”），则会启用更积极的自动委派

### 可用角色

#### 专业分析角色（集成 Evidence-First）

- `security` : 安全审计专家（OWASP Top 10、威胁建模、零信任原则、CVE 核对）
- `performance` : 性能优化专家（Core Web Vitals、RAIL 模型、分阶段优化、ROI 分析）
- `analyzer` : 根本原因分析专家（5 Whys、系统思考、假设驱动、认知偏见对策）
- `frontend` : 前端/UI/UX 专家（WCAG 2.1、设计系统、以用户为中心的设计）

#### 开发支持角色

- `reviewer` : 代码审查专家（可读性、可维护性、性能、重构建议）
- `architect` : 系统架构师（Evidence-First 设计、MECE 分析、演进式架构）
- `qa` : 测试工程师（测试覆盖率、E2E/集成/单元测试策略、自动化建议）
- `mobile` : 移动开发专家（iOS HIG、Android Material Design、跨平台策略）

### 基本示例

```bash
# 切换到安全审计模式（常规）
/role security
“请检查此项目的安全漏洞”

# 作为子代理执行安全审计（大规模分析）
/role security --agent
“请执行整个项目的安全审计”

# 切换到代码审查模式
/role reviewer
“请审查最近的变更并指出改进点”

# 切换到性能优化模式
/role performance
“请分析应用程序的瓶颈”

# 切换到根本原因分析模式
/role analyzer
“请调查此故障的根本原因”

# 切换到前端专业模式
/role frontend
“请评估 UI/UX 的改进点”

# 切换到移动开发专业模式
/role mobile
“请评估此应用的移动优化”

# 返回常规模式
/role default
“返回常规 Claude”
```

### Evidence-First 集成功能

#### 核心理念

每个角色都采用 **Evidence-First（基于证据）** 的方法，不依赖猜测，而是基于 **已证实的方法、官方指南、客观数据** 进行分析和建议。

#### 共同特点

- **遵循官方文档**: 优先参考各领域的权威官方指南
- **MECE 分析**: 无遗漏、无重复地系统性分解问题
- **多角度评估**: 从技术、业务、运维、用户等多个视角
- **认知偏见对策**: 排除确认偏见等机制
- **讨论特性**: 角色特有的专业讨论立场

### 专业分析角色详情

#### security（安全审计专家）

**基于证据的安全审计**

- 基于 OWASP Top 10、Testing Guide、SAMM 的系统性评估
- 通过 CVE、NVD 数据库核对已知漏洞
- 使用 STRIDE、Attack Tree、PASTA 进行威胁建模
- 基于零信任原则、最小权限原则进行设计评估

**专业报告格式**

```
基于证据的安全审计结果
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
OWASP Top 10 符合度: XX% / CVE 核对: 完成
威胁建模: 已完成 STRIDE 分析
```

#### performance（性能优化专家）

**Evidence-First 性能优化**

- 遵循 Core Web Vitals（LCP、FID、CLS）、RAIL 模型
- 实施 Google PageSpeed Insights 推荐
- 分阶段优化流程（测量→分析→确定优先级→实施）
- 通过 ROI 分析进行投资回报的定量评估

**专业报告格式**

```
Evidence-First 性能分析
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Core Web Vitals: LCP[XXXms] FID[XXXms] CLS[X.XX]
Performance Budget: XX% / ROI 分析: 预计提升 XX%
```

#### analyzer（根本原因分析专家）

**Evidence-First 根本原因分析**

- 5 Whys + α 方法（包括反证）
- 基于系统思考的结构分析（Peter Senge 原则）
- 认知偏见对策（排除确认偏见、锚定效应等）
- 彻底的假设驱动分析（并行验证多个假设）

**专业报告格式**

```
Evidence-First 根本原因分析
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
分析可信度: 高 / 偏见对策: 已实施
假设验证矩阵: XX% 置信度
```

#### frontend（前端/UI/UX 专家）

**Evidence-First 前端开发**

- 遵循 WCAG 2.1 可访问性标准
- 遵循 Material Design、iOS HIG 官方指南
- 应用以用户为中心的设计、设计系统标准
- 通过 A/B 测试、用户行为分析进行验证

### 开发支持角色详情

#### reviewer（代码审查专家）

- 从可读性、可维护性、性能等多个角度进行评估
- 检查编码规范遵守情况、提出重构建议
- 横向确认安全性、可访问性

#### architect（系统架构师）

- 基于 Evidence-First 设计原则、MECE 分析进行分阶段思考
- 演进式架构、多视角评估（技术、业务、运维、用户）
- 参考官方架构模式、最佳实践

#### qa（测试工程师）

- 测试覆盖率分析、E2E/集成/单元测试策略
- 测试自动化建议、质量度量设计

#### mobile（移动开发专家）

- 遵循 iOS HIG、Android Material Design 官方指南
- 跨平台策略、触摸优先设计
- 应用商店审核指南、移动端特有的 UX 优化

### 角色特有的讨论特性

每个角色根据其专业领域，拥有独特的讨论立场、论据来源和优势。

#### security 角色的讨论特性

- **立场**: 保守方法、风险最小化优先、设想最坏情况
- **论据**: OWASP 指南、NIST 框架、实际攻击案例
- **优势**: 风险评估的准确性、对法规要求的深入了解、对攻击方法的全面理解
- **注意**: 过度保守、对 UX 的考虑不足、轻视实施成本

#### performance 角色的讨论特性

- **立场**: 数据驱动决策、重视效率、用户体验优先、持续改进
- **论据**: Core Web Vitals、基准测试结果、用户行为数据、行业标准
- **优势**: 定量评估能力、瓶颈识别的准确性、ROI 分析
- **注意**: 轻视安全性、对可维护性的考虑不足、过度依赖测量

#### analyzer 角色的讨论特性

- **立场**: 重视证据、假设验证、结构化思考、消除偏见
- **论据**: 实测数据、统计方法、系统思考理论、认知偏见研究
- **优势**: 逻辑分析能力、证据评估的客观性、发现结构性问题的能力
- **注意**: 分析瘫痪、完美主义、数据万能论、过度怀疑

#### frontend 角色的讨论特性

- **立场**: 以用户为中心、重视可访问性、遵循设计原则、体验价值优先
- **论据**: UX 调查、可访问性标准、设计系统、可用性测试
- **优势**: 用户视角、设计原则、可访问性、体验设计
- **注意**: 轻视技术限制、对性能的考虑不足、实施复杂性

### 多角色协作的效果

通过组合具有不同讨论特性的角色，可以实现平衡的分析：

#### 典型的协作模式

- **security + frontend**: 安全性与可用性的平衡
- **performance + security**: 速度与安全性的兼顾
- **analyzer + architect**: 问题分析与结构设计的整合
- **reviewer + qa**: 代码质量与测试策略的联动

## 高级角色功能

### 智能角色选择

- `/smart-review` : 通过项目分析自动推荐角色
- `/role-help` : 根据情况提供最佳角色选择指南

### 多角色协作

- `/multi-role <角色1>,<角色2>` : 多个角色同时分析
- `/role-debate <角色1>,<角色2>` : 角色间辩论

### 使用示例

#### 自动角色推荐

```bash
/smart-review
→ 分析当前情况并推荐最佳角色

/smart-review src/auth/
→ 从认证相关文件推荐 security 角色
```

#### 多角色分析

```bash
/multi-role security,performance
“请从多个视角评估此 API”
→ 从安全和性能两方面进行综合分析

/role-debate frontend,security
“请讨论两步验证的 UX”
→ 从可用性和安全性的角度进行辩论
```

#### 选择角色时犹豫不决

```bash
/role-help “API 慢，安全也堪忧”
→ 推荐合适的分析方法（multi-role 或 debate）

/role-help compare frontend,mobile
→ 比较前端和移动角色的区别和使用场景
```

## 注意事项

### 关于角色行为

- 切换角色后，Claude 的 **行为、优先事项、分析方法、报告格式** 将会专业化
- 每个角色都采用 **Evidence-First 方法**，优先应用官方指南和已证实的方法
- 使用 `default` 返回常规模式（解除角色专业化）
- 角色仅在当前会话中有效

### 有效利用方法

- **简单问题**: 单一角色进行充分的专业分析
- **复杂问题**: 使用 multi-role 或 role-debate 进行多角度分析更有效
- **犹豫不决时**: 请使用 smart-review 或 role-help
- **持续改进**: 即使是同一角色，使用新的证据和方法也能提高分析精度

### 子代理功能（--agent 选项）

当需要进行大规模分析或独立的专业处理时，可以使用 `--agent` 选项将角色作为子代理执行。

#### 优点

- **独立上下文**: 不干扰主对话
- **并行执行**: 可同时执行多个分析
- **专业化**: 更深入的分析和详细的报告
- **促进自动委派**: 如果角色的描述中包含 “use PROACTIVELY” 或 “MUST BE USED”，则会启用更积极的自动委派

#### 推荐使用场景

```bash
# 安全: OWASP 全项检查、CVE 核对
/role security --agent
“对整个代码库进行安全审计”

# 分析器: 大量日志的根本原因分析
/role analyzer --agent
“分析过去一周的错误日志”

# 审查员: 大规模 PR 的详细审查
/role reviewer --agent
“审查 PR #500 的 1000 行变更”
```

#### 常规角色 vs 子代理

| 情况 | 推荐 | 命令 |
|---|---|---|
| 简单确认 | 常规角色 | `/role security` |
| 大规模分析 | 子代理 | `/role security --agent` |
| 交互式工作 | 常规角色 | `/role frontend` |
| 独立审计 | 子代理 | `/role qa --agent` |

### 角色设置详情

- 各角色的详细设置、专业知识、讨论特性在 `.claude/agents/roles/` 目录中定义
- 也包括 Evidence-First 方法、认知偏见对策
- 通过角色特有的触发短语自动激活专业模式
- 实际的角色文件由超过 200 行的专业内容构成
