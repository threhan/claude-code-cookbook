## PR 审查

通过对 Pull Request 的系统性审查，确保代码质量和架构的健全性。

### 使用方法

```bash
# PR 的全面审查
gh pr view 123 --comments
“请系统地审查此 PR，并从代码质量、安全性和架构的角度提供反馈”

# 安全专项审查
gh pr diff 123
“请重点审查安全风险和漏洞”

# 从架构角度审查
gh pr checkout 123 && find . -name "*.js" | head -10
“请从分层、依赖关系和 SOLID 原则的角度评估架构”
```

### 基本示例

```bash
# 代码质量的数值评估
find . -name "*.js" -exec wc -l {} + | sort -rn | head -5
“请评估代码的复杂度、函数大小和重复度，并指出改进点”

# 安全漏洞检查
grep -r "password\|secret\|token" . --include="*.js" | head -10
“请检查机密信息泄露、硬编码和认证绕过的风险”

# 架构违规检测
grep -r "import.*from.*\.\./\.\." . --include="*.js"
“请评估分层违规、循环依赖和耦合度问题”
```

### 评论分类体系

```
🔴 critical.must: 致命问题
├─ 安全漏洞
├─ 数据一致性问题
└─ 系统故障风险

🟡 high.imo: 高优先级改进
├─ 功能不全的风险
├─ 性能问题
└─ 可维护性大幅下降

🟢 medium.imo: 中优先级改进
├─ 提高可读性
├─ 改善代码结构
└─ 提高测试质量

🟢 low.nits: 轻微指摘
├─ 统一风格
├─ 修正拼写错误
└─ 添加注释

🔵 info.q: 提问/提供信息
├─ 确认实现意图
├─ 设计决策的背景
└─ 分享最佳实践
```

### 审查角度

#### 1. 代码正确性

- **逻辑错误**: 边界值、Null 检查、异常处理
- **数据一致性**: 类型安全、验证
- **错误处理**: 覆盖范围、适当的处理

#### 2. 安全性

- **认证/授权**: 适当的检查、权限管理
- **输入验证**: SQL 注入、XSS 对策
- **机密信息**: 禁止日志输出、加密

#### 3. 性能

- **算法**: 时间复杂度、内存效率
- **数据库**: N+1 查询、索引优化
- **资源**: 内存泄漏、缓存利用

#### 4. 架构

- **分层**: 依赖方向、适当的分离
- **耦合度**: 紧耦合、接口利用
- **SOLID 原则**: 单一职责、开闭、依赖倒置

### 审查流程

1. **事前确认**: PR 信息、变更差异、相关 Issue
2. **系统性检查**: 安全性 → 正确性 → 性能 → 架构
3. **建设性反馈**: 具体的改进建议和代码示例
4. **跟进**: 确认修改、CI 状态、最终批准

### 有效的评论示例

**安全问题**

```markdown
**critical.must.** 密码以明文形式保存

```javascript
// 修复建议
const bcrypt = require('bcrypt');
const hashedPassword = await bcrypt.hash(password, 12);
```

为防止安全风险，必须进行哈希处理。

```

**性能改进**
```markdown
**high.imo.** 出现 N+1 查询问题

```javascript
// 改进建议: Eager Loading
const users = await User.findAll({ include: [Post] });
```

可以大幅减少查询次数。

```

**架构违规**
```markdown
**high.must.** 出现分层违规

领域层直接依赖于基础设施层。
请使用依赖倒置原则引入接口。
```

### 注意事项

- **建设性语气**: 采用协作而非攻击性的沟通方式
- **具体建议**: 不仅指出问题，还提供解决方案
- **确定优先级**: 按 Critical → High → Medium → Low 的顺序处理
- **持续改进**: 将审查结果知识化
