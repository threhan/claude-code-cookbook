## 检查提示

这是一份用于评估和改进面向 AI Agent 的提示质量的综合性最佳实践集合。它将实际提示改进过程中积累的知识系统化，涵盖了消除歧义、信息整合、加强强制性、追踪系统、持续改进等所有重要方面。

### 使用方法

```bash
# 检查提示文件的质量
cat your-prompt.md
/check-prompt
“请检查此提示的质量并提出改进建议”
```

### 选项

- 无：分析当前文件或选定的文本
- `--category <name>`：仅检查特定类别 (structure/execution/restrictions/quality/roles/improvement)
- `--score`：仅计算质量得分
- `--fix`：自动提出修复建议以修正检测到的问题
- `--deep`：深度分析模式（重点检查歧义、信息分散、强制性）

### 基本示例

```bash
# 评估整个提示的质量
cat devin/playbooks/code-review.md
/check-prompt
“请从 6 个方面评估此提示的质量，并指出问题点和改进建议”

# 深度分析模式
/check-prompt --deep
“请重点检查歧义、信息分散、强制力不足，并提出根本性的改进建议”

# 检查特定类别
/check-prompt --category structure
“请从结构和明确性的角度检查此提示”

# 检测并修正歧义表达
/check-prompt --fix
“请检测歧义表达并提出明确的修改建议”
```

## 核心设计原则

### 原则 1：完全消除解释的余地

- **绝对禁止**：“原则上”、“推荐”、“如果可能”、“根据情况”、“酌情判断”
- **必须使用**：“必须”、“绝对”、“严格遵守”、“无一例外”、“强制”
- **例外条件**：用数值严格限定（“仅限以下 3 个条件”、“除这两种情况外”）

### 原则 2：信息的战略性整合

- 将相关重要信息完全整合到一个部分
- 在执行检查清单中总结整体情况
- 彻底消除引用的循环或分散

### 原则 3：构建分阶段的强制力

- 🔴 (执行停止级别) → 🟡 (质量重要) → 🟢 (推荐事项) 的明确层级
- 从推荐级别逐步提升至必须级别
- 明确违反时的影响程度和处理方法

### 原则 4：确保可追溯性

- 所有执行结果都可记录和验证
- 从技术上防止虚假报告
- 成功/失败的客观判断标准

### 原则 5：反馈驱动的改进

- 从实际失败案例中学习
- 持续验证有效性
- 自动检测新模式

## 🚨 致命性问题模式（需要立即修正）

### ❌ 级别 1：致命性歧义（执行停止级别）

- **可产生多种解释的指示**：“酌情判断”、“根据情况”、“原则上”
- **模糊的例外条件**：“特殊情况下”、“根据需要”
- **主观的判断标准**：“适当地”、“充分地”、“尽可能”
- **未定义的重要概念**：“标准的”、“一般的”、“基本的”

### ❌ 级别 2：结构性缺陷（质量重要级别）

- **信息分散**：相关重要信息散布在 3 个以上的位置
- **循环引用**：部分 A→B→C→A 的无限循环
- **矛盾的指示**：不同部分存在相互矛盾的指示
- **执行顺序不明确**：依赖关系不清晰的步骤

### ❌ 级别 3：质量下降（推荐改进级别）

- **无法验证**：成功/失败的判断标准不明
- **难以自动化**：依赖于人工主观判断的设计
- **难以维护**：更新时影响范围无法预测的结构
- **难以学习**：新人需要花费时间才能理解的复杂性

## 🎯 经过验证的改进方法

### ✅ 分阶段强化方法

1. **现状分析**：对问题进行分类、确定优先级、评估影响
2. **优先解决致命性问题**：最优先完全解决级别 1 的问题
3. **分阶段实施**：不一次性进行所有变更，而是以可验证的单位进行
4. **效果测量**：对改进前后进行定量比较
5. **持续监控**：确认改进效果的持久性

### ✅ 消除歧义的实践方法

```markdown
# ❌ 改进前（模糊）

“指出的问题，原则上应作为内联评论写在 GitHub 上相应的变更处”

# ✅ 改进后（明确）

“指出的问题，必须作为内联评论写在 GitHub 上相应的变更处。例外情况仅限于第 3.3 节中定义的 3 个条件”
```

### ✅ 信息整合的实践方法

```markdown
# ❌ 改进前（分散）

第 2.1 节：“使用必要的 6 个部分”
第 3.5 节：“📊 综合评估，📋 指出的问题...”
第 4.2 节：“禁止删除部分”

# ✅ 改进后（整合）

执行检查清单：
□ 10. 发布摘要评论（使用必要的 6 个部分）
🔴 必要的 6 个部分：1) 📊 综合评估 2) 📋 按分类汇总指出的问题 3) ⚠️ 主要担忧 4) ✅ 值得称赞之处 5) 🎯 结论 6) 🤖 AI 审查质量的自我评估
❌ 绝对禁止：删除、添加或重命名部分
```

### ✅ 追踪系统的实施模式

```bash
# 严格追踪执行结果
POSTED_COMMENTS=0
FAILED_COMMENTS=0
TOTAL_COMMENTS=0

# 记录每个操作的结果
if [ $? -eq 0 ]; then
    echo "✅ 成功: $OPERATION" >> /tmp/execution_log.txt
    POSTED_COMMENTS=$((POSTED_COMMENTS + 1))
else
    echo "❌ 失败: $OPERATION" >> /tmp/execution_log.txt
    FAILED_COMMENTS=$((FAILED_COMMENTS + 1))
fi

# 防止虚假报告
if [ $POSTED_COMMENTS -ne $REPORTED_COMMENTS ]; then
    echo "🚨 警告：报告数量与实际发布数量不符"
    exit 1
fi
```

## 📈 质量得分计算（改进版）

### 综合得分计算

```
基本得分 = Σ(各类别得分 × 分值) / 100

致命性问题扣分：
- 级别 1 问题：-20 分/个
- 级别 2 问题：-10 分/个
- 级别 3 问题：-5 分/个

加分项：
- 支持自动化：+5 分
- 实现学习功能：+5 分
- 经过验证的改进案例：+5 分

最终得分 = 基本得分 + 加分 - 扣分
```

### 质量级别判定

```
95-100 分：世界顶级水平 (可作为行业标准推荐)
90-94 分：优秀 (可用于生产环境)
80-89 分：良好 (稍加改进即可用于生产环境)
70-79 分：一般 (需要改进)
60-69 分：需要改进 (需要大幅修正)
50-59 分：需要大幅修正 (需要根本性地重新审视)
49 分以下：禁止使用 (需要完全重新设计)
```

## 🔧 实践性的改进流程

### 阶段 1：诊断与分析（1-2 天）

1. **掌握整体结构**：可视化部分构成、信息流、依赖关系
2. **检测歧义**：全部提取出有解释余地的表达
3. **分析信息分散**：映射相关信息的分散模式
4. **评估强制力**：评估推荐/必须的分类及其有效性
5. **确认可追溯性**：评估执行结果的记录和验证功能

### 阶段 2：确定优先级与计划（半天）

1. **致命性分类**：对级别 1-3 的问题进行分类并评估影响
2. **确定改进顺序**：考虑相互依赖关系，确定最佳顺序
3. **分配资源**：优化改进效果与成本的平衡
4. **评估风险**：预测改进时的副作用和兼容性问题

### 阶段 3：分阶段实施（2-5 天）

1. **解决级别 1 问题**：完全消除致命性歧义
2. **实施信息整合**：战略性地集中分散的信息
3. **加强强制力**：从推荐逐步提升至必须
4. **实施追踪系统**：自动记录和验证执行结果的功能
5. **强化模板**：明确必要元素并强制遵守

### 阶段 4：验证与调整（1-2 天）

1. **功能测试**：确认所有变更点的功能
2. **集成测试**：确认整个系统的一致性
3. **性能测试**：确认执行效率和响应
4. **可用性测试**：在实际使用场景中进行验证

### 阶段 5：运维与监控（持续）

1. **效果测量**：对改进前后进行定量比较
2. **持续监控**：及早发现质量下降
3. **收集反馈**：提取实际运维中的问题点
4. **持续优化**：持续的改进循环

### 经验教训与最佳实践

#### 成功因素

1. **分阶段方法**：不一次性进行所有变更，而是以可验证的单位进行
2. **数据驱动**：不依赖主观判断，而是基于实测数据进行改进
3. **持续监控**：定期确认改进效果的持久性
4. **重视反馈**：积极收集实际用户的意见

#### 失败规避策略

1. **过度追求完美**：达到 90 分后即投入使用，通过持续改进达到 100 分
2. **一次性变更的危险性**：大规模变更必须分阶段实施
3. **向后兼容性**：最大限度地减少对现有工作流的影响
4. **文档不足**：详细记录和共享所有变更
