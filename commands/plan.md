## 计划

启动实施前的计划制定模式，以制定详细的实施策略。通过在代码实施前制定结构化计划，支持高效的开发。

### 使用方法

```bash
# 请求 Claude 进入计划模式
“请为 [实施内容] 制定实施计划”
```

### 基本示例

```bash
# 新功能的实施计划
“请为用户认证功能制定实施计划”

# 系统设计的计划
“请为微服务拆分制定实施计划”

# 重构计划
“请为遗留代码的重构制定计划”
```

### 与 Claude 协作

```bash
# 复杂功能实施
“请为聊天功能制定实施计划。包括 WebSocket、实时通知、历史记录管理”

# 数据库设计
“请为电商网站的数据库设计制定计划。包括商品、订单、用户管理”

# API 设计
“请为 GraphQL API 的实施制定计划。包括认证、缓存、速率限制”

# 基础设施设计
“请为 Docker 化制定实施计划。包括开发环境、生产环境、CI/CD”
```

### 计划模式的特点

**自动启动**

- 检测到实施任务时自动启动计划模式
- 可以通过“制定实施计划”等关键字明确启动

**结构化规格书**

- 需求定义（用户故事、验收标准）
- 设计书（架构、数据设计、UI 设计）
- 实施计划（任务分解、进度跟踪、质量保证）
- 风险分析和对策

**审批流程**

- 使用 `exit_plan_mode` 工具提出计划
- **重要**: 无论工具的返回值如何，都必须等待用户的明确批准
- 禁止在未经批准的情况下开始实施
- 可以修改和调整计划
- 只有在批准后才能使用 TodoWrite 开始任务管理

### 详细示例

```bash
# 复杂系统实施
“请为在线支付系统制定实施计划。包括 Stripe 集成、安全性、错误处理”

# 前端实施
“请为 React 仪表板制定实施计划。包括状态管理、组件设计、测试”

# 后端实施
“请为 RESTful API 制定实施计划。包括认证、验证、日志记录”

# DevOps 实施
“请为 CI/CD 流水线制定实施计划。包括测试自动化、部署、监控”
```

### 三阶段工作流

#### 阶段 1: 需求

- **用户故事**: 明确功能的​​目的和价值
- **验收标准**: 定义完成条件和质量标准
- **约束/前提条件**: 整理技术和时间上的约束
- **确定优先级**: 分类为必须有/可以有

#### 阶段 2: 设计

- **架构设计**: 系统配置和技术选型
- **数据设计**: 模式、API 规格、数据流
- **UI/UX 设计**: 屏幕配置和操作流程
- **风险分析**: 潜在问题和对策

#### 阶段 3: 实施

- **任务分解**: 细分为可实施的单元
- **进度跟踪**: 使用 TodoWrite 进行状态管理
- **质量保证**: 测试策略和验证方法
- **审批流程**: 使用 exit_plan_mode 提出计划并等待明确批准

### 注意事项

**适用范围**

- 计划模式最适合复杂的实施任务
- 对于简单的修改或小规模的变更，请使用常规的实施方式
- 推荐用于需要 3 个以上步骤的工作或新功能开发

**技术约束**

- 不信任 `exit_plan_mode` 工具的返回值
- 审批流程由用户的明确意图决定
- 与 CLI 的 plan mode 功能不同

**执行注意事项**

- 严禁在批准前开始实施
- 提出计划后必须等待用户响应
- 出现错误时提出替代方案
