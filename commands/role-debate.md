## 角色辩论

具有不同专业知识的角色进行辩论，权衡利弊，从而得出最佳解决方案的命令。

### 使用方法

```bash
/role-debate <角色1>,<角色2> [议题]
/role-debate <角色1>,<角色2>,<角色3> [议题]
```

### 基本示例

```bash
# 安全 vs 性能的权衡
/role-debate security,performance
“关于 JWT 令牌的有效期设置”

# 可用性 vs 安全性的平衡
/role-debate frontend,security
“关于两步验证的 UX 优化”

# 技术选型辩论
/role-debate architect,mobile
“关于选择 React Native 还是 Flutter”

# 三方辩论
/role-debate architect,security,performance
“关于是否应该微服务化”
```

### 辩论的基本原则

#### 建设性辩论的心得

- **相互尊重**: 尊重其他角色的专业知识和观点
- **基于事实**: 基于数据和证据进行辩论，而非情绪化的反驳
- **以解决问题为导向**: 旨在找到更好的解决方案，而非为了批评而批评
- **重视实现**: 提出考虑了可行性的建议，而非理想化的理论

#### 论据的质量要求

- **官方文档**: 引用标准、指南、官方文档
- **实证案例**: 具体引用成功和失败的案例
- **定量评估**: 尽可能使用数值和指标进行比较
- **时间序列考虑**: 评估短期、中期和长期的影响

#### 辩论伦理

- **诚实**: 承认自己专业领域的局限性
- **开放**: 对新的信息和观点保持灵活性
- **透明**: 明确判断的依据和前提条件
- **负责**: 提及建议的实施风险

### 辩论过程

### 阶段 1: 表明初步立场

每个角色从专业角度独立发表意见

- 提出推荐方案
- 明确依据的标准和文档
- 说明预期的风险和挑战
- 定义成功指标

### 阶段 2: 相互辩论与反驳

角色之间进行交叉辩论

- 对其他角色的提议进行建设性的反驳
- 指出被忽略的观点
- 明确权衡利弊
- 提出替代方案

### 阶段 3: 寻求妥协点

探索可行的解决方案

- 评估各观点的权重
- 探讨双赢的解决方案
- 考虑分阶段实施的方法
- 探讨风险缓解措施

### 阶段 4: 综合结论

确定最终的推荐方案

- 达成共识的解决方案
- 实施路线图
- 成功指标和衡量方法
- 未来的审查要点

### 输出格式示例

### 两角色辩论的情况

```
角色辩论: Security vs Performance
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

议题: JWT 令牌的有效期设置

Security 角色的主张:
“推荐使用 15 分钟的短期有效期”

依据:
- 遵循 OWASP JWT Security Cheat Sheet
- 最大限度地减少令牌泄露时的损失时间窗口
- 限制攻击者的可用时间

担忧:
- 长期有效期会使攻击风险呈指数级增长
- 合规性要求（金融领域）要求短期有效期

成功指标:
- 安全事件发生率 < 0.1%
- 平均攻击检测时间 < 5 分钟

Performance 角色的反驳:
“推荐使用 2 小时的有效期”

依据:
- 参考 Google OAuth 2.0 Best Practices
- 避免因频繁重新认证而增加服务器负载
- 最大限度地减少对用户体验（工作中断）的影响

担忧:
- 每 15 分钟重新认证一次会使 API 负载增加 8 倍
- 移动环境下会频繁出现连接中断

成功指标:
- API 响应时间保持在 < 200ms
- 服务器 CPU 使用率 < 60%

相互辩论:
Security → Performance:
“与服务器负载相比，安全漏洞造成的业务损失更为严重。
例如：Equifax 事件造成了 7 亿美元的损失”

Performance → Security:
“可以通过刷新令牌机制实现两全其美。
通过后台更新，可以在不影响 UX 的情况下确保安全”

Security → Performance:
“刷新令牌也是攻击目标。前提是实现得当”

Performance → Security:
“建议分阶段实施。常规操作 30 分钟，敏感操作 15 分钟”

寻求妥协点:
共同理解:
- 需要兼顾用户体验和安全性
- 根据风险级别灵活应对
- 现实地考虑实施和运维成本

双赢要素:
- 利用刷新令牌机制
- 分阶段引入基于风险的认证
- 通过自动注销功能进行补充

综合结论:
“30 分钟有效期 + 刷新令牌 + 基于风险的认证”

实施细节:
1. 访问令牌: 30 分钟有效期
2. 刷新令牌: 7 天有效期
3. 高风险操作: 15 分钟后强制重新认证
4. 30 分钟无操作自动注销

分阶段实施:
第 1-2 周: 实现基本的 30 分钟令牌
第 3-4 周: 添加刷新令牌机制
第 2 个月: 引入基于风险的认证

成功指标:
- 安全性: 事件发生率 < 0.1%
- 性能: API 负载增长率 < 20%
- UX: 用户满意度 > 85%

未来审查:
- 3 个月后: 评估实际的攻击模式和负载情况
- 6 个月后: 考虑迁移到更完善的基于风险的认证
```

### 三角色辩论的情况

```
角色辩论: Architect vs Security vs Performance
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

议题: 是否应该微服务化

Architect 角色的主张:
“推荐分阶段微服务化”
依据: 明确领域边界、独立部署、技术选型的自由度

Security 角色的担忧:
“服务间通信的安全性变得复杂”
依据: API 网关、mTLS、分布式认证的管理成本

Performance 角色的担忧:
“网络通信导致延迟增加”
依据: 内部 API 调用导致的 N+1 问题、分布式事务

三方辩论:
Architect → Security: “可以通过 API 网关集中管理来控制”
Security → Architect: “存在单点故障的风险”
Performance → Architect: “服务拆分的粒度很重要”
...（继续辩论）

综合结论:
“基于领域驱动设计的分阶段拆分 + 安全优先的设计”
```

### 按角色的辩论特性

#### 🔒 Security 角色

```yaml
辩论立场:
  - 保守方法 (风险最小化)
  - 重视规则遵守 (对偏离标准持谨慎态度)
  - 设想最坏情况 (攻击者视角)
  - 重视长期影响 (作为技术债务的安全性)

典型论点:
  - “安全性 vs 便利性” 的权衡
  - “必须满足合规性要求”
  - “攻击成本 vs 防御成本的比较”
  - “彻底保护隐私”

论据来源:
  - OWASP 指南
  - NIST 框架
  - 行业标准 (ISO 27001, SOC 2)
  - 实际攻击案例和统计数据

辩论中的优势:
  - 风险评估的准确性
  - 对法规要求的了解
  - 对攻击方法的理解

需要注意的偏见:
  - 过度保守 (阻碍创新)
  - 对 UX 的考虑不足
  - 轻视实施成本
```

#### ⚡ Performance 角色

```yaml
辩论立场:
  - 数据驱动决策 (基于测量)
  - 重视效率 (成本效益优化)
  - 用户体验优先 (重视感知速度)
  - 持续改进 (分阶段优化)

典型论点:
  - “性能 vs 安全性”
  - “优化成本 vs 效果的投资回报”
  - “当前 vs 未来的可扩展性”
  - “用户体验 vs 系统效率”

论据来源:
  - Core Web Vitals 指标
  - 基准测试结果和统计数据
  - 用户行为影响数据
  - 行业性能标准

辩论中的优势:
  - 定量评估能力
  - 识别瓶颈
  - 优化方法的知识

需要注意的偏见:
  - 轻视安全性
  - 对可维护性的考虑不足
  - 过早优化
```

#### 🏗️ Architect 角色

```yaml
辩论立场:
  - 重视长期视角 (考虑系统演进)
  - 追求平衡 (整体优化)
  - 分阶段变更 (风险管理)
  - 遵循标准 (优先使用经过验证的模式)

典型论点:
  - “短期效率 vs 长期可维护性”
  - “技术债务 vs 开发速度”
  - “微服务 vs 单体”
  - “采用新技术 vs 稳定性”

论据来源:
  - 架构模式集
  - 设计原则 (SOLID, DDD)
  - 大规模系统案例
  - 技术演进趋势

辩论中的优势:
  - 全局视野能力
  - 设计模式的知识
  - 预测长期影响

需要注意的偏见:
  - 过度泛化
  - 对新技术持保守态度
  - 对实施细节理解不足
```

#### 🎨 Frontend 角色

```yaml
辩论立场:
  - 以用户为中心的设计 (UX 最优先)
  - 包容性方法 (考虑多样性)
  - 重视直观性 (学习成本最小化)
  - 可访问性标准 (遵循 WCAG)

典型论点:
  - “可用性 vs 安全性”
  - “设计统一 vs 平台优化”
  - “功能性 vs 简洁性”
  - “性能 vs 丰富的体验”

论据来源:
  - UX 研究和可用性测试结果
  - 可访问性指南
  - 设计系统标准
  - 用户行为数据

辩论中的优势:
  - 代表用户视角
  - 设计原则的知识
  - 可访问性要求

需要注意的偏见:
  - 对技术限制理解不足
  - 轻视安全要求
  - 低估性能影响
```

#### 📱 Mobile 角色

```yaml
辩论立场:
  - 平台特定 (考虑 iOS/Android 差异)
  - 适应上下文 (移动中、单手操作)
  - 资源限制 (电池、内存、网络)
  - 遵循应用商店规则 (审核指南)

典型论点:
  - “原生 vs 跨平台”
  - “离线支持 vs 实时同步”
  - “电池效率 vs 功能性”
  - “平台统一 vs 优化”

论据来源:
  - iOS HIG / Android Material Design
  - App Store / Google Play 指南
  - 移动 UX 研究
  - 设备性能统计

辩论中的优势:
  - 理解移动端特有的限制
  - 了解平台差异
  - 触摸界面设计

需要注意的偏见:
  - 对 Web 平台理解不足
  - 轻视服务器端限制
  - 对桌面环境考虑不足
```

#### 🔍 Analyzer 角色

```yaml
辩论立场:
  - 重视证据 (数据优先)
  - 假设验证 (科学方法)
  - 结构化思考 (系统思考)
  - 消除偏见 (追求客观性)

典型论点:
  - “相关性 vs 因果关系”
  - “对症下药 vs 根本解决”
  - “区分假设 vs 事实”
  - “短期症状 vs 结构性问题”

论据来源:
  - 实测数据和日志分析
  - 统计方法和分析结果
  - 系统思考理论
  - 认知偏见研究

辩论中的优势:
  - 逻辑分析能力
  - 证据评估的客观性
  - 发现结构性问题

需要注意的偏见:
  - 分析瘫痪 (行动力不足)
  - 完美主义 (轻视实用性)
  - 数据万能论
```

### 辩论质量检查清单

#### 论据质量

- [ ] 引用了官方文档/标准
- [ ] 提供了具体的案例/数据
- [ ] 明确区分了推测和事实
- [ ] 注明了信息来源

#### 辩论的建设性

- [ ] 准确理解了对方的提议
- [ ] 进行了逻辑而非情绪化的反驳
- [ ] 提出了替代方案
- [ ] 探索了双赢的可能性

#### 实施可能性

- [ ] 考虑了技术可行性
- [ ] 估算了实施成本/时间
- [ ] 探讨了分阶段实施的可能性
- [ ] 提出了风险缓解措施

#### 综合性

- [ ] 考虑了对其他领域的影响
- [ ] 追求了整体优化
- [ ] 包括了长期视角
- [ ] 设定了可衡量的成功指标

### 注意事项

- 辩论可能需要一些时间（越复杂的主题时间越长）
- 三个以上角色的辩论可能会发散
- 最终决策请用户参考辩论结果自行做出
- 对于紧急性高的问题，请先考虑使用 single role 或 multi-role
