## 错误修复

从错误消息中识别根本原因，并提出经过验证的解决方案。

### 使用方法

```bash
/fix-error [选项]
```

### 选项

- 无：标准错误分析
- `--deep`：深度分析模式（包括依赖关系和环境因素）
- `--preventive`：重视预防措施的分析
- `--quick`：仅提供可立即应用的修复

### 基本示例

```bash
# 标准错误分析
npm run build 2>&1
/fix-error
“请分析构建错误并提供修复方法”

# 深度分析模式
python app.py 2>&1
/fix-error --deep
“请分析错误的根本原因，包括环境因素”

# 重视立即修复
cargo test 2>&1
/fix-error --quick
“请提供可立即应用的修复方法”

# 重视预防措施
./app 2>&1 | tail -50
/fix-error --preventive
“请提供错误的修复方法和未来的预防措施”
```


### 错误分析的优先级

#### 紧急度：高（必须立即处理）

- **应用程序停止**：崩溃、无限循环、死锁
- **数据丢失风险**：数据库错误、文件损坏
- **安全漏洞**：认证失败、权限错误、注入
- **生产环境影响**：部署失败、服务中断

#### 🟡 紧急度：中（建议尽早处理）

- **性能问题**：内存泄漏、延迟、超时
- **部分功能失常**：特定功能错误、UI 故障
- **开发效率下降**：构建错误、测试失败

#### 🟢 紧急度：低（计划性处理）

- **警告消息**：弃用、lint 错误
- **仅限开发环境**：仅在本地环境出现的问题
- **未来风险**：技术债务、可维护性问题

### 分析过程

#### 阶段 1：收集错误信息

```bash
🔴 必须执行：
- 完整获取错误消息
- 检查堆栈跟踪
- 确定发生条件（可复现性）

🟡 建议尽早执行：
- 收集环境信息（操作系统、版本、依赖关系）
- 最近的变更历史（git log、最近的提交）
- 检查相关日志

🟢 建议补充执行：
- 系统资源状况
- 网络状态
- 外部服务状态
```

#### 阶段 2：根本原因分析

1. **整理表面症状**
   - 错误消息的准确内容
   - 发生时间和模式
   - 确定影响范围

2. **识别深层原因**
   - 应用 5 Whys 分析
   - 追踪依赖关系
   - 检查环境差异

3. **验证假设**
   - 创建最小可复现代码
   - 执行隔离测试
   - 缩小原因范围

#### 阶段 3：实施解决方案

```bash
🔴 立即处理（热修复）：
- 抑制症状的最小修复
- 应用临时解决方法
- 准备紧急部署

🟡 根本解决：
- 针对原因的根本性修复
- 添加测试用例
- 更新文档

🟢 实施预防措施：
- 加强错误处理
- 设置监控和警报
- 改进 CI/CD 流水线
```

### 按错误类型分析方法

#### 编译/构建错误

```bash
# TypeScript 类型错误
必须检查（高）：
- tsconfig.json 的配置
- 类型定义文件（.d.ts）是否存在
- import 语句的准确性

# Rust 生命周期错误
必须检查（高）：
- 所有权的移动
- 引用的有效期
- 可变性的冲突
```

#### 运行时错误

```bash
# Null/Undefined 引用
必须检查（高）：
- 缺少可选链
- 初始化时机
- 等待异步处理完成

# 内存相关错误
必须检查（高）：
- 获取堆转储
- 分析 GC 日志
- 检测循环引用
```

#### 依赖关系错误

```bash
# 版本冲突
必须检查（高）：
- lock 文件的一致性
- peer dependencies 的要求
- 传递性依赖

# 模块解析错误
必须检查（高）：
- NODE_PATH 设置
- 路径别名设置
- 符号链接
```

### 注意事项

- **绝对禁止**：仅凭部分错误消息判断、未经验证就应用 Stack Overflow 的解决方案
- **例外情况**：仅在以下三种情况下允许临时解决方法
  1. 生产环境的紧急应对（必须在 24 小时内根本解决）
  2. 外部服务故障（在等待恢复期间的替代方案）
  3. 已知的框架 bug（等待修复版本发布）
- **推荐**：最优先识别根本原因，避免表面修复

### 最佳实践

1. **完整收集信息**：从头到尾检查错误消息
2. **确认可复现性**：最优先创建最小可复现代码
3. **分阶段方法**：从小处着手进行修复和验证
4. **文档化**：记录解决过程以共享知识

#### 常见陷阱

- **治标不治本**：忽略根本原因的表面修复
- **过度泛化**：将特定情况的解决方案广泛应用
- **省略验证**：不确认修复后的副作用
- **知识孤岛**：不将解决方法文档化
